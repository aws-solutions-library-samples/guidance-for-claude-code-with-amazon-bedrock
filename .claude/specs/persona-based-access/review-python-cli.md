### Critical
- **[`deploy.py:1317-1333` `_resolve_issuer_host` + `deploy.py:1801-1809` `_resolve_oidc_config`] Persona trust-condition issuer namespace is WRONG for Auth0 and Azure → those IdPs' persona users hard-deny (silent total failure for v1-supported providers).** _(Lead-confirmed CRITICAL, 2026-06-16; fix task #30.)_
  The persona role's STS trust condition keys on `<issuer_host>:<groups_claim>`, where `issuer_host` = `_resolve_oidc_config(...)` scheme-stripped and `rstrip("/")`. For `AssumeRoleWithWebIdentity`, the condition-key prefix is the **registered OIDC-provider URL with scheme removed** and must equal it EXACTLY (slash/path preserved), per `issuer-url-format.md` (Auth0 slash REQUIRED; Azure `/v2.0`, NO slash).
  - **Auth0:** the auth stack registers the provider as `https://${Auth0Domain}/` (trailing slash — `bedrock-auth-auth0.yaml:76`), so the condition-key prefix is `company.auth0.com/`. `_resolve_issuer_host` strips the slash → emits `company.auth0.com:groups`. `test_deploy_personas.py::test_strips_trailing_slash_for_auth0` asserts the (incorrect) strip as *intended*. The `ForAnyValue:StringEquals` never matches → **all Auth0 persona users assume no role (hard-deny)**. The `_resolve_issuer_host` docstring even cites `issuer-url-format.md`, then `.rstrip("/")` strips exactly the slash that rule requires.
  - **Azure:** provider registered as `https://login.microsoftonline.com/${tenant}/v2.0`; `_resolve_issuer_host` returns `provider_domain` scheme-stripped with **no** canonicalization to that `/v2.0` form (and `provider_domain` may be a raw URL or a bare GUID — the auth path calls `_extract_azure_tenant_id` precisely because it isn't normalized). No Azure test exists. Same hard-deny risk.
  - Okta/Google/generic are fine (provider URL host == `provider_domain`, no path).
  FR-2.7 lists Auth0 + Azure as v1-supported; this is a silent total failure for them, not cosmetic. **Fix (task #30):** derive `issuer_host` from the auth template's exact OIDC-provider `Url` form (preserve Auth0 trailing slash; build the canonical `login.microsoftonline.com/<tenant>/v2.0` for Azure, no slash), not from `provider_domain`; add Auth0 + Azure regression tests.

- **[`config.py` (Profile) + `init.py` (`_gather_personas`, `wizard_fields`) + `deploy.py:1641`] FR-6.1 account-total budget is unreachable through the supported flow.**
  `_deploy_budgets_stack` reads `getattr(profile, "account_budget_amount_usd", None)` and `budgets_template.render_budgets_stack` fully supports an account-wide budget, but `account_budget_amount_usd` is **not** a `Profile` dataclass field, is **not** collected by the init wizard, and is **not** in `wizard_fields`. So the `getattr` always returns `None` and the account-total budget can never be configured — FR-6.1 ("plus one account-total budget") is partially undeliverable. Per-persona budgets (`budget_amount_usd` per persona) work. Spec §4.1 explicitly lists `account_budget_amount_usd` as a top-level sibling. **Fix (task #31):** add the field to `Profile`, collect it in `_gather_personas`, and include it in `wizard_fields`. _(Lead-confirmed WARNING, 2026-06-16.)_

- **[`destroy.py:27-40` DESTROYABLE_STACKS + `deploy.py:1670-1710` `_deploy_persona_dashboard`] FR-9.5 teardown gap: the inline persona-dashboard stack is created but never destroyed.** `[via feature-dev:code-reviewer, verified]`
  `_deploy_persona_dashboard` deploys a real CloudFormation stack named `{identity_pool_name}-persona-dashboard` (when personas exist and the template is present), but `persona-dashboard` is **not** in `DESTROYABLE_STACKS`. `ccwb destroy` (full or `persona`) therefore never tears it down — the line-158 skip-guard that lists `"persona-dashboard"` is dead code because the stack is never in the iteration. The deploy↔destroy parity test (`test_destroy_stacks.py`) does not catch this: its `_deployable_stack_types()` regex only matches `stacks_to_deploy.append((...))`, and the dashboard is deployed via a direct method call, so it's invisible to the regex. FR-9.5 says destroy must tear down persona resources cleanly. Small blast radius (a dashboard; no IAM/data), hence Warning not Critical. **Fix (task #32):** either add `"persona-dashboard"` to `DESTROYABLE_STACKS` (before `persona`), or add explicit best-effort dashboard-stack deletion in `destroy.py` analogous to `_delete_persona_inference_profiles`. ### Note on a dismissed finding
The plugin reviewer flagged `StringEquals` on `aws:RequestedRegion` with a `CommaDelimitedList` Ref as an always-deny CRITICAL. **Dismissed as a false positive:** the shipped, production `bedrock-auth-*.yaml` templates use the identical pattern (`StringEquals: {aws:RequestedRegion: !Ref AllowedBedrockRegions}` with a `CommaDelimitedList`), and CFN expands a `CommaDelimitedList` `Ref` into a JSON array in the rendered policy — IAM `StringEquals` against a multi-valued condition is an OR match. The renderer correctly mirrors the established pattern; `ForAnyValue:StringEquals` is not required. The plugin's issuer-host finding overclaimed Okta (Okta's provider URL is host-only, so the host match is correct) — captured here in narrower, accurate form (Auth0/Azure only). Its `effective_auth_type` "dataclasses forbid attribute assignment" claim is incorrect (assignment works; test proves it).
